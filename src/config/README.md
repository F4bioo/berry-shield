# Configuration Modules - Berry Shield

This directory contains the central configuration catalog, runtime defaults, normalization helpers, and Web UI schema source for Berry Shield.

---

## `catalog.ts`

This module is the single source of truth for Berry Shield configurable product values.

### What it does
1.  **Defines**: Configurable defaults, enum contracts, minimum values, descriptions, and `uiHints`.
2.  **Builds**: The runtime default config consumed by `defaults.ts`.
3.  **Builds**: The Web `configSchema` and `uiHints` consumed by `schema.ts`.
4.  **Answers**: The maintenance question "where do I change this configurable value?".

### When to edit
- Add a new configurable field.
- Change an existing configurable default.
- Change enum values, descriptions, minimum values, or Web UI hints.
- Keep this file aligned with `types/config.ts`.

---

## `defaults.ts`

This module exposes the runtime default config derived from `catalog.ts`.

### What it does
1.  **Exports**: The effective default values used when plugin config is missing or partial.
2.  **Anchors**: Shared defaults used by config merging and runtime behavior.
3.  **Keeps**: Runtime consumers on top of the same source used by the Web contract.

### When to edit
- Usually do not edit this file directly for value changes.
- Edit `catalog.ts` when the product default changes.
- Keep this file aligned with `types/config.ts` only if the exported facade shape changes.

---

## `schema.ts`

This module is the Web schema facade derived from `catalog.ts`.

### What it does
1.  **Exports**: The plugin `configSchema` used by the OpenClaw Web UI.
2.  **Exports**: The plugin `uiHints` used to improve Web form rendering.
3.  **Synchronizes**: Provides manifest serialization helpers used by `scripts/sync-manifest.mjs`.
4.  **Keeps**: The manifest publisher isolated from the internal catalog implementation.

### When to edit
- Usually do not edit this file directly for config field changes.
- Edit `catalog.ts` when the Web contract needs a new field or metadata change.
- Keep this file aligned only if schema export boundaries or manifest helpers change.

---

## `utils.ts`

This module normalizes user config and merges it with runtime defaults.

### What it does
1.  **Parses**: Partial or untrusted plugin config input.
2.  **Normalizes**: Values into the runtime contract expected by Berry Shield.
3.  **Merges**: User config with `DEFAULT_CONFIG`.
4.  **Guards**: Falls back safely when values are invalid or missing.

### When to edit
- Add parsing/normalization for a new config field.
- Tighten validation or fallback behavior.
- Keep merge behavior aligned with `types/config.ts` and `defaults.ts`.

---

## `wrapper.ts`

This module contains config wrapper helpers used by CLI flows that read and write plugin config paths.

### What it does
1.  **Reads**: Config values through stable plugin config paths.
2.  **Writes**: Config updates in CLI-facing flows.
3.  **Bridges**: Berry Shield CLI commands with the OpenClaw config structure.

### When to edit
- Add or change CLI config path handling.
- Adjust CLI write/read behavior for new plugin settings.

---

## Recommended Workflow

When adding or changing a configurable feature:

1.  Update `src/types/config.ts` with the contract shape.
2.  Update `src/config/catalog.ts` with the configurable default and schema metadata.
3.  Update `src/config/utils.ts` with merge/normalization logic.
4.  Update CLI commands if the field must be configurable from CLI.
5.  Run `npm run typecheck`, `npm test`, or `npm run build` to publish the generated Web contract automatically before validation.
6.  Add or update tests to prevent drift and regressions.

For internal, non-configurable runtime constants:
- Edit `src/constants.ts`, not `catalog.ts`.

---

## Classification Rule

- `catalog.ts`: product defaults and published config contract.
- `constants.ts`: technical constants, protocol constants, IDs, and stable enum value lists not owned by product defaults.
- `defaults.ts`: derived runtime facade.
- `schema.ts`: derived Web/schema facade.

---

## Common Tasks

### Change a default value

1.  Update the default in `src/config/catalog.ts`.
2.  Update merge behavior in `src/config/utils.ts` only if fallback or validation rules also changed.
3.  Run `npm run typecheck`, `npm test`, or `npm run build`.
4.  Use `npm run plugin:manifest:sync` only when you want a standalone sync without the normal QA/build commands.

### Add a new config field

1.  Add the field to `src/types/config.ts`.
2.  Add the field definition to `src/config/catalog.ts`.
3.  Add parsing and normalization to `src/config/utils.ts`.
4.  Update CLI commands if the field must be configurable from CLI.
5.  Run `npm run typecheck`, `npm test`, or `npm run build`.
6.  Add or update contract and regression tests.

### Change Web copy or labels only

1.  Update descriptions or `uiHints` in `src/config/catalog.ts`.
2.  Run `npm run typecheck`, `npm test`, or `npm run build`.
3.  Confirm only generated manifest sections changed.

---

## Drift Prevention

The Web UI contract is not maintained manually in `openclaw.plugin.json`.

- `src/config/catalog.ts` is the source of truth for configurable defaults and generated manifest sections.
- `src/config/schema.ts` is the derived facade consumed by the sync script.
- `scripts/sync-manifest.mjs` publishes `configSchema` and `uiHints` automatically before `typecheck`, `test`, and `build`.
- `__tests__/plugin-manifest-schema-contract.test.ts` protects against drift.
